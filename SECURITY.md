# Security Policy

## Supported Versions

| Version | Supported |
|---------|-----------|
| `main`  | Yes       |

Only the latest commit on `main` is actively supported. If you find a vulnerability, please report it against the current `main` branch.

## Reporting a Vulnerability

**Do not open a public GitHub issue for security vulnerabilities.**

Please report security issues privately to:

**[INSERT SECURITY CONTACT]**
(e.g., a private email address, GitHub private vulnerability report, or other secure channel)

Include as much of the following information as possible so we can understand the nature and scope of the issue:

- Type of issue (e.g., auth bypass, secret exposure, injection, SSRF)
- Full path of the source file(s) related to the vulnerability
- Location of the affected source code (tag/branch/commit or direct URL)
- Step-by-step instructions to reproduce the issue
- Proof-of-concept or exploit code (if possible)
- Impact assessment — what an attacker could accomplish

You should receive an initial acknowledgment within **72 hours**. We aim to release a patch or mitigation within **7 days** for critical issues.

## Scope

This policy covers the `better-prode` application code. It does not cover third-party dependencies directly — please report those to the upstream maintainer.

## Disclosure Policy

We follow coordinated disclosure. We ask that you give us a reasonable window to address the issue before any public disclosure.
